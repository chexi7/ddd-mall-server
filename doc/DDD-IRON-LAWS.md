# DDD 铁律（来自 cap4j）

以下规则从 cap4j 框架中提取，作为 DDD 设计的硬性约束：

## 聚合相关

1. **聚合之间不相互引用** — 聚合是独立的一致性边界，不应直接持有其他聚合的引用，只能通过 ID 关联。
2. **聚合中可以包含多个实体** — 一个聚合是实体和值对象的集合，内部可以有多个实体，但对外只暴露聚合根。
3. **一个实体不能出现在多个聚合中** — 实体只能属于一个聚合，跨越聚合边界的同一业务概念应通过 ID 引用而非共享实体。

## 命令相关

4. **一个命令处理器只操作一个聚合** — 命令的处理逻辑只修改一个聚合实例，保证事务边界清晰，避免跨聚合的一致性问题。
5. **聚合仅可以在命令处理器中进行操作** — 聚合的状态变更只能通过命令处理器触发，其他任何入口（查询、事件处理器等）不允许直接修改聚合。

## 查询相关

6. **一个查询只从一个聚合中获取数据** — 查询不应跨聚合组合数据，避免引入聚合间的耦合；跨聚合的数据组合在应用层或读模型中完成。

## 事件相关

7. **一个事件可以有多个事件处理器** — 一个聚合产生的事件可以被多个处理器消费，实现业务流程的解耦与编排。
8. **事件处理器不操作聚合** — 事件处理器不直接修改聚合状态，只能通过发出命令间接影响聚合，保证状态变更的唯一入口是命令处理器。
9. **一个事件处理器只发出一个命令** — 每个事件处理器只产生一个命令，避免复杂的编排逻辑混入事件处理器，流程编排应由 Saga/流程管理器负责。

## 入口相关

10. **一个 Controller 的 Action 只允许发出一个命令** — Controller 的每个 Action（方法）只发出一个命令，不做业务逻辑编排，编排职责交给事件驱动的流程。
11. **一个 Job 的任务只允许发出一个命令** — 定时任务的每次执行只发出一个命令，与 Controller Action 同理，保持入口的简洁性。
